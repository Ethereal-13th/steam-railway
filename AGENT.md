## 1. 任务定义

你正在实现一个 Minecraft Fabric 模组：`steam_railway`。

这个模组的产品核心不是“更快的运输”，而是：

- 玩家亲手建造自己的列车和线路
- 玩家可以乘坐列车旅行、看风景
- 列车在玩家不在附近时仍能继续逻辑运行

任何实现决策都必须优先服务这三个目标。

---

## 2. 绝对约束

### 2.1 产品约束

- 不修改原版矿车、原版铁轨、原版实体行为
- 不依赖 Create 的应力系统
- 火车结构只来自玩家显式强力胶连接
- 首版目标场景是 `2-10` 人朋友服

### 2.2 工程约束

- 先做 Fabric，不为 NeoForge 牺牲首版速度
- 列车逻辑状态不依赖可见实体持续存在
- 服务端权威，客户端只做表现和插值
- 不过早引入多线程
- 所有玩家可见文本必须走 Minecraft 本地化系统，不允许新增硬编码英文提示
- 至少同时维护 `zh_cn` 和 `en_us` 两套语言资源；新增文本时必须两边一起补齐
- 命令反馈、物品名、方块名、实体名、GUI 文本、调试提示统一使用 `Text.translatable(...)` 或等价本地化入口
- 不把 Phase 1 三大难题同时开工：
  - 轨道编辑器
  - Create 风格组装结构
  - 自由站立移动平台

### 2.3 首版禁止事项

在 `P0a-P0d` 完成前，不要实现这些：

- 多轨距
- 道岔
- 多节车厢挂钩
- 燃料系统
- 重量系统
- 红石联动扩展
- 高保真粒子/动画
- 高级 GUI 打磨

必须按下面顺序推进，除非用户明确改范围：

1. `P0a` 工程骨架、注册、调试命令

## 4. 模块边界

建议结构以 [docs/蒸汽铁路-实施计划.md](./docs/蒸汽铁路-实施计划.md) 为准：

- `rail/`
  只负责轨道图、路径采样、轨道方块与铺轨工具
- `train/runtime/`
  只负责 `TrainState`、速度、路径推进、可见实体刷新
- `train/entity/`
  只负责火车实体表现层
- `train/schedule/`
  只负责时刻表和司机方块
- `contraption/`
  只负责显式胶水、列车结构组装/解散、座位映射
- `persistence/`
  统一处理世界存档数据
- `net/`
  统一处理 C2S/S2C 包

禁止把这些层混在一起：

- 轨道图模块里直接写列车乘客逻辑
- Train runtime 里直接解析胶水结构
- Client 渲染代码里维护权威状态

## 5. 实现优先级

### 5.1 第一优先级

先证明这件事成立：

> 一列车在未加载区块时仍然按逻辑继续运行，并在玩家返回后出现在正确位置。

没有这一点，项目的独特价值不成立。

### 5.2 第二优先级

再证明这件事成立：

> 玩家可以不用命令，自己铺一条线路，并让列车沿线路运行。

### 5.3 第三优先级

最后再证明：

> 玩家可以坐上去体验旅行感，而不是只看它在动。

## 6. 代码风格与工程原则

- 优先写清晰、可替换的代码，不追求抽象炫技
- 先建最小可验证版本，再叠加复杂体验
- 一个提交尽量只解决一个明确问题
- 行为变化和结构重构不要混在同一提交
- 重要状态结构必须能序列化、反序列化、回归测试
- 凡是会展示给玩家的文本，都要优先写成可翻译 key，再落到 `assets/steam_railway/lang/*.json`

推荐每个核心模块都至少有：

- 一个入口 service
- 一个纯逻辑 solver / assembler
- 一个数据结构定义
- 一个测试文件

---

## 7. 测试要求

至少覆盖这些回归点：

- 路径存档后重载不丢
- 列车状态存档后重载不丢
- 区块卸载后列车进度继续推进
- 玩家返回后列车位置不倒退
- 胶水不误吸收未粘合建筑
- 解散还原不复制方块、不吞方块

建议优先写的测试：

- `RailPathSamplerTest`
- `TrainMotionSolverTest`
- `SRSaveDataTest`
- `GlueGraphTraversalTest`
- `ScheduleStateTest`

如果新功能无法写自动测试，必须至少留下明确的手测步骤。

新增涉及玩家文本的功能时，手测至少覆盖：

- `简体中文` 下文本可读、语义正确
- `English (US)` 下文本存在，不回退成 key
- 命令成功、失败、边界错误三类提示都能正确切换语言

## 8. 代理执行协议

每次开始工作前，先做这三步：

1. 读 PRD 和实施计划，确认当前阶段
2. 说明本次只处理哪个里程碑或子模块
3. 检查当前改动是否跨越多个阶段，如果跨越就收缩范围

每次结束工作时，输出这四项：

1. 本次完成了什么
2. 还剩什么
3. 当前阻塞点是什么
4. 下一步最合理的任务是什么

---

## 9. Definition of Done

一个阶段只能在同时满足以下条件后算完成：

- 对应目标功能可运行
- 对应状态可保存和恢复
- 有基础测试或手测清单
- 不破坏上一阶段已完成能力
- 文档中该阶段的验收项已逐条核对
