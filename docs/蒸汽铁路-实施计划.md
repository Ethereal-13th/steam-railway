## 1. 目标与范围

### 1.1 本文档回答的问题

这份文档不重复 PRD 的产品愿景，专门回答四件事：

1. 首版先做什么，不做什么
2. 代码应该怎么分层，避免后面返工
3. 每个阶段完成的定义是什么
4. 开始编码后，任务如何拆给自己逐步推进

### 1.2 工程目标

首个可实施目标不是“完整铁路模组”，而是：

> 在 Fabric 上做出一个可保存、可重载、可在未加载区继续逻辑运行的火车原型，并在此基础上逐步叠加轨道铺设、列车组装、时刻表、乘坐体验。

### 1.3 工程原则

- `Boring by default`：优先选 Minecraft/Fabric 常规做法，不为了“更优雅”引入过早复杂度。
- `状态和表现分离`：列车逻辑状态不依赖实体一直存在。
- `先跑通，再做完整体验`：Ghost Train Proof 是第一硬门槛。
- `显式粘合，不做自动扫描`：火车结构只来自玩家明确的胶水连接。
- `服务端权威`：列车位置、速度、路径进度、时刻表进度都由服务端决定。
- `玩家文本必须可本地化`：所有玩家可见文本默认进入 Minecraft 原生本地化系统，至少同步维护 `zh_cn` 与 `en_us`。
- `不接受硬编码英文提示`：命令反馈、物品名、方块名、实体名、GUI 文本、调试提示统一用翻译 key 和语言资源文件维护。

## 2. 首版切分

### 2.1 推荐实施顺序

不要直接从 PRD 的 “Phase 1 最小可玩版本” 开始写。真正建议的落地顺序是：

1. `P0a` 仓库脚手架 + 注册体系 + 调试命令
2. `P0b` RailNetwork 原型 + 两点路径生成
3. `P0c` TrainState + 服务端 tick 推进 + 存档恢复
4. `P0d` 可见 TrainEntity 同步 + 区块重载后位置正确
5. `P1a` 真实轨道方块 + 铺轨工具 + 轨道图注册
6. `P1b` 基座 + 显式强力胶连接 + 实体化/解散
7. `P1c` 手动驾驶 + 时刻表 + 自动驾驶
8. `P1d` 坐垫乘坐 + 低速自由走动

### 2.2 为什么这样切

PRD 当前把三个难题放在同一层：

- 轨道编辑器
- Create 风格移动结构
- 玩家在移动平台上自由走动

如果并行做，定位 bug 会非常慢。应先把“火车逻辑是否可靠”单独做通，再叠加“结构化列车”，最后再做“移动平台上的玩家体验”。

### 2.3 明确不阻塞首版的内容

这些功能存在于 PRD，但不允许阻塞开工：

- 多轨距
- 道岔
- 多节车厢挂钩
- 信号灯
- 高级 UI 打磨
- 高保真动画和粒子
- 完整红石联动

## 3. 系统架构

### 3.1 总体分层

```text
┌──────────────────────────────────────────────┐
│ Client                                       │
│  - HUD                                       │
│  - 轨道预览                                 │
│  - 胶水选择框                               │
│  - 插值显示的火车实体                       │
└──────────────────────────────────────────────┘
                     │ packets
                     ▼
┌──────────────────────────────────────────────┐
│ Server                                       │
│  - 命令/交互输入                             │
│  - RailNetworkService                        │
│  - TrainRuntimeService                       │
│  - ScheduleService                           │
│  - ContraptionAssemblyService                │
│  - SaveData                                  │
└──────────────────────────────────────────────┘
                     │
                     ▼
┌──────────────────────────────────────────────┐
│ World Data                                   │
│  - RailNetwork                               │
│  - TrainState                               │
│  - ContraptionState                         │
│  - Station registry                         │
└──────────────────────────────────────────────┘
```

### 3.2 核心原则

- `RailNetwork` 是铁路权威数据，不从世界轨道逐格推断运行路径。
- `TrainState` 是列车权威状态，实体只是表现层。
- `ContraptionState` 是列车结构快照，静止时可写回方块，运行时跟随火车实体。
- 区块卸载后保留 `TrainState`，不保留实体作为逻辑前提。

## 6. 关键流程

### 6.1 列车运行流程

```text
Server tick
  -> TrainRuntimeService.tickAll()
     -> 读取 TrainState
     -> 用 TrainMotionSolver 推进 distance/speed/status
     -> 检查是否到站/是否切边/是否停车
     -> 若附近有玩家或区块已加载
        -> 更新/生成 TrainEntity
     -> 写回 TrainState
```

### 6.2 区块卸载后恢复流程

```text
区块卸载
  -> TrainEntity 可消失
  -> TrainState 保留
  -> 服务端继续 tick TrainState

玩家返回
  -> TrainSpawnManager 检查该列车是否需要可见实体
  -> 按 TrainState 当前位置生成 TrainEntity
  -> 客户端收到快照并插值到正确位置
```

### 6.3 列车结构组装流程

```text
玩家放置基座
  -> 使用强力胶创建显式连接
  -> ContraptionAssembler 从基座出发遍历 glueLinks
  -> 生成 ContraptionState
  -> TrainEntity 激活时引用 ContraptionState
```

### 6.4 手动驾驶流程

```text
客户端按住 W/S
  -> 发送 C2S 驾驶输入
  -> Server 更新 TrainState.targetSpeed
  -> TrainMotionSolver 按加速度/减速度推进
  -> 周期性同步位置快照
```

## 8. 首版不该过早做的事

### 8.1 不要现在做

- 多线程列车求解
- 多节车厢挂钩
- 道岔与复杂寻路
- 重量系统
- 燃料系统
- 多轨距
- 高保真轮子/悬挂动画

### 8.2 原因

这些都不是首版价值核心，而且都会显著增加定位成本。现在最重要的是：  
`一列车 -> 一条线 -> 一个世界状态 -> 可保存 -> 可回来`

## 10. 性能与风险

### 10.1 主要风险

1. 自由走动时玩家抖动、穿模、掉落
2. 列车结构过大导致碰撞计算开销过高
3. 区块重载后实体位置校正出现明显跳变
4. 铺轨工具在长路径预览时掉帧
5. 解散还原时方块放置冲突

### 10.2 风险对策

- 自由走动只在 `P1d` 做，且先限定低速、少玩家
- `ContraptionState` 预生成简化碰撞盒，不逐方块每 tick 重新算
- 快照同步只发附近列车，客户端插值
- 铺轨预览限制最大长度，当前 PRD 已定 `50 格`
- 解散前先做空间合法性校验，失败则拒绝解散并提示

### 10.3 性能红线

- 单列车 `64` 方块为首版默认上限
- 同时运行列车先按 `<= 8` 设计
- 首版只针对 `2-10` 人朋友服调优
